    PCIDSS Compliance

    PCI DSS - Top 3 issues¶

    Payment Card Industry Data Security Standard is a document produced by PCI Security council, an organisation formed by credit card scheme companies, joined by financial institutions, retailers and professional companies.

    Security professionals have been calling for improving security for some time, stating best practices. Now there is clear business case for implementing these well known best practices and organisation will need to allocate budgets (CAPEX and OPEX) to run programmes to be and stay compliant. PCI DSS will not go away, it is likely to address more security controls in the future.

    The PCI DSS contains 6 objectives detailed in 12 requirements:

    • Build and Maintain a Secure Network

      • Requirement 1: Install and maintain a firewall configuration to protect cardholder data
      • Requirement 2: Do not use vendor-supplied defaults for system passwords and other security parameters

    • Protect Cardholder Data

      • Requirement 3: Protect stored cardholder data
      • Requirement 4: Encrypt transmission of cardholder data across open, public networks

    • Maintain a Vulnerability Management Program

      • Requirement 5: Use and regularly update anti-virus software
      • Requirement 6: Develop and maintain secure systems and applications

    • Implement Strong Access Control Measures

      • Requirement 7: Restrict access to cardholder data by business need-to-know
      • Requirement 8: Assign a unique ID to each person with computer access
      • Requirement 9: Restrict physical access to cardholder data

    • Regularly Monitor and Test Networks

      • Requirement 10: Track and monitor all access to network resources and cardholder data
      • Requirement 11: Regularly test security systems and processes

    • Maintain an Information Security Policy

      • Requirement 12: Maintain a policy that addresses information security

    Quick assessment¶

    While to comply with this standard an organisation must fulfil all requirements, clearly there are some harder to address and some with bigger impact on overall security of an organisation. I will try to do quick assessment of each requirement, before going into more details.

    Requirement 1: Install and maintain a firewall configuration to protect cardholder data¶

    Not surprisingly this is an easy target. Most of the organisation do have a firewall protecting internal systems from attacks from the Internet. On the other hand, segregation within the internal network is something not very popular and organisations will struggle with this requirement. As they might do with requirements related to documentation. My feeling is that average organisation will find approximately 50% compliance with all requirements here.

    Requirement 2: Do not use vendor-supplied defaults for system passwords and other¶

    This should be an easy one as well but is largely overlooked on almost any organisation. This covers hardening, secure administrators access and securing wireless networks. Organisations with legacy systems will have limited options to comply with this part of PCI DSS.

    Requirement 3: Protect stored cardholder data¶

    Tough one. Unless an organisation have new systems it will struggle with this part of PCI DSS. This is the one that has the biggest impact on the security and is also one of the most important requirements in the standard. The best approach is to minimise storage of the data or to eliminate it completely. I guess an average organisation will comply with 10% of this part of PCI DSS.

    Requirement 4: Encrypt transmission of cardholder data across open, public networks¶

    This requirement seems to an easy one but legacy applications might not support it. How do you think TJ Maxx got hacked? Retailers, if using wireless devices might not be able to comply and will need to upgrade the technology.

    Requirement 5: Use and regularly update anti-virus software Show me an organisation that does not run an antivirus. The question is: is it installed, configured and managed properly and targets all necessary systems? However, I think this will be like 80% compliance when going a gap analysis. Probably also the easiest to fix.

    Requirement 6: Develop and maintain secure systems and applications¶

    Some of these requirements are easy to implement and some are not. Organisations will struggle with segregation of development, test and production environments while change management is probably well established. Interesting is introduction of web application firewalls. I guess 40% compliance with most issues around vulnerability management and segregation of development systems.

    Requirement 7: Restrict access to cardholder data by business need-to-know¶

    Mostly process oriented requirement. management can see this as easy one but it might be tricky and might require redesign of internal processes. Despite I think an average organisation should be 50% compliant.

    Requirement 8: Assign a unique ID to each person with computer access Legacy systems will be the biggest obstacle here so banks and retailers will be hit very hard. New technologies make this one an easy fix with big impact. Maybe it is time to implement SSO?

    Requirement 9: Restrict physical access to cardholder data¶

    Security within information systems is of no use if computers or paper files are compromised. An average data centre should be compliant but other areas might be tricky to secure. My guess is 20% compliance but relatively easy and cheap fix.

    Requirement 10: Track and monitor all access to network resources and cardholder data¶

    It is easy to log but difficult to analyse so this requirement will be tricky to achieve. Yet I think organisation are 60% there.

    Requirement 11: Regularly test security systems and processes This seems like an easy one but will be difficult to achieve and certainly costly, mainly an OPEX type of expense as this is a BAU activity. While organisation might have done external testing, internal is rarely performed, network intrusion detection/prevention systems are expensive and file integrity monitoring systems are a dream of security managers. Companies like Qualys love this one.

    Requirement 12: Maintain a policy that addresses information security¶

    How many organisation do have Information Security Policy, maybe based on ISO 27001, in place, approved by management and appropriately communicated (lived)? This might seem like an easy requirement but will require cross-functional resources to address all aspects. Also service providers, if touching card holder data, have to adhere to the standard.

    Analysis¶

    Clearly, looking back at these 12 requirements we can identify these top 3 hard to fix:

    Requirement 3: Protect stored cardholder data¶

    Let's have a look at an average online system. It consists of: ◦ web server ◦ application server ◦ database server

    Is any server logging the activity, perhaps credit card number? If so, and business requires it, this information must be stored encrypted. Also data in a database must be encrypted. Encryption can be performed by an application, or database system and analysis should be done to see which solution is more appropriate. There are technical solution on the market that provide off the shelf secure storage.

    Another issue might be the key management. It is nice to have data encrypted but if the key is easily available or weak the encryption bears minimal value. Key management systems are available off the shelf to address this.

    This online system is surely not the only system in the organisation so other applications will need to be looked at as well, if they access credit card data. The worst scenario is to have several repositories of credit card data as this is difficult to secure, and sure expensive. it is much better to consolidate data into one system, segregated, secured with all security controls. You can call it the vault. Credit card data should never leave this vault, only be added to it or deleted.

    Next, customer agents use the system to service customer calling to the call centre and buy over the phone. What do they do if the system is not available. Do they use another system, write information on the paper? An organisation might need to change its processes with regards to off-line processing to be compliant.

    Requirement 6: Develop and maintain secure systems and applications¶

    Patch management! This requirement closely complements 11.2. Organisations will find it difficult to patch systems in timely manner, purely due to testing needed and other priorities. Some patches might involve application changes.

    Segregation of development, test and production environment is also interesting one. SOX requires logical segregation of access, i.e. developers and testers not having access to production environment. In addition PCI DSS goes little further by requiring segregation on network and system level. Also not using real data for development and testing is hard to achieve, this has been practice for years. UK Data Protection Act does not require this.

    Web Application Firewall (WAF) is new concept that will likely be incorporated into existing firewalls. Some firewalls already contain such modules however as any firewall, this requires testing, maintenance and changes as the application changes. The standard also does not specify detailed requirements on WAF, but I suggest using requirement 1 as a baseline.

    Requirement 11: Regularly test security systems and processes¶

    How many times has an organisation build a system, BISed it and forgot about it. External and internal penetration testing, vulnerability scanning is ongoing activity that will require internal or external resources. network intrusion detection/prevention systems are nightmare to setup and operate and in a typical architecture will need to be in several locations to analyse all relevant traffic. This will include: ◦ Internet zone ◦ DMZ zone with Web server ◦ MZ zone with application server ◦ MZ zone with database server

    Conclusion¶

    Security professionals have been calling for improving security for some time, stating best practices. Now there is clear business case for implementing these well known best practices and organisation will need to allocate budgets (CAPEX and OPEX) to run programmes to be and stay compliant. PCI DSS will not go away, it is likely to address more security controls in the future.
